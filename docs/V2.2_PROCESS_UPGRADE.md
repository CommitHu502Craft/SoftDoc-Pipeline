# V2.2 流程升级：章程 -> 规格 -> 验证 -> 冻结

## 1. 升级目标
- 从“仅项目名”升级为“章程先行”。
- 代码前必须有可执行规格，避免模糊文案直生代码。
- 文档前必须有运行验证报告。
- 形成可追溯冻结包，支撑申报材料反向证明。
- 通过规格确认机制，确保“规格先审后生代码”。

## 2. 新增关键产物
- `output/<项目名>/project_charter.json`
- `output/<项目名>/project_executable_spec.json`
- `output/<项目名>/spec_review_status.json`
- `output/<项目名>/spec_review_guide.md`
- `output/<项目名>/runtime_verification_report.json`
- `output/<项目名>/freeze_package/manifest.json`
- `output/<项目名>/freeze_package/artifact_hashes.json`
- `output/<项目名>/freeze_package/reproducibility_report.json`
- `output/<项目名>/freeze_package/change_log.md`
- `output/<项目名>/<项目名>_freeze_package.zip`

## 3. 章程字段（必填）
- `business_scope`: 业务边界
- `user_roles`: 角色列表（至少2个）
- `core_flows`: 核心流程（至少1条，每条至少2步）
- `non_functional_constraints`: 非功能约束（至少1条）
- `acceptance_criteria`: 验收标准（至少1条）

## 4. 默认流水线
`plan -> spec -> html -> screenshot -> code -> verify -> document -> pdf -> freeze`

说明：
- `plan` 阶段会校验章程，不通过即阻断。
- `spec` 阶段产出结构化规格并校验。
- `spec` 阶段会刷新规格评审状态：digest 变更后自动回 `pending`。
- `code/verify/document/pdf/freeze` 阶段要求规格为 `approved`（默认自动确认，也支持手动确认）。
- `verify` 阶段在文档前执行本地 smoke 与流程回放。
- `freeze` 阶段计算产物哈希并生成归档包。

## 5. LLM 调用影响
- 新增步骤 `spec/verify/freeze` 均为本地计算，不新增 LLM 调用。
- 仍需 LLM 的步骤仅为 `plan/html/code`，与旧版本一致。

## 6. 入口行为差异
- `Qt`：支持“AI 草拟章程 + 人工确认”；规格默认自动确认，可配置为人工确认。
- `Web`：可查看与确认规格；默认自动确认，缺规格时可直接完整流程运行。
- `API`：提供 `spec-review` 状态/确认接口；实现阶段门禁默认自动确认，可配置关闭。

## 7. V2.3 补充能力（本次增强）

### 7.1 Web 章程前置 + 批量补全
- 新增接口：
  - `GET /api/projects/{id}/charter`
  - `PUT /api/projects/{id}/charter`
  - `POST /api/projects/{id}/charter/draft`
  - `POST /api/projects/batch-charter/draft`
- 前端支持项目内编辑章程、AI补全章程、批量补全章程。

### 7.2 自动修复并继续
- 新增接口：
  - `POST /api/projects/{id}/self-heal-run`
  - `POST /api/projects/batch-self-heal-run`
- 行为：缺章程自动草拟；缺 `plan/spec` 自动插入对应步骤并继续流水线。

### 7.3 申报前风险预检
- 新增接口：
  - `GET /api/projects/{id}/submission-risk`
  - `POST /api/projects/{id}/submission-risk/check`
- 提交队列 `start` 前自动执行风险预检，不通过直接拦截并标记失败。

### 7.4 同质化语义闸门
- `plan/spec` 后执行跨项目语义相似检测（实体/流程/API）。
- 超阈值自动重写规格语义并同步 `project_plan.executable_spec`，再生成评审工件。

### 7.5 冻结包证据链增强
- 新增产物：
  - `freeze_package/api_replay_evidence.json`
  - `freeze_package/key_task_logs.json`
  - `freeze_package/artifact_hash_chain.json`
  - `freeze_package/development_change_record.json`

### 7.6 LLM 预算与稳定性
- 新增 `llm_budget`（分阶段预算 + 失败计数）。
- `DeepSeekClient` 支持 prompt 缓存、失败后自动切换备用模型（若配置 `fallback_model(s)`）。

## 8. V3.1 增量（Skill 驱动 + 硬门禁）

> 详细说明见：`docs/V3.1_RUNTIME_SKILL_GATE_GUIDE.md`

### 8.1 Skill Orchestration Layer
- 新增统一产物：
  - `ui_skill_profile.json`
  - `ui_blueprint.json`
  - `screenshot_contract.json`
- HTML/截图/文档阶段统一消费这三类契约，不再各自猜测页面结构。

### 8.2 运行时规则图与合规校验
- 新增：
  - `runtime_rule_graph.json`
  - `skill_compliance_report.json`
  - `skill_policy_decision_report.json`
  - `skill_autorepair_report.json`
- 合规失败后只重写失败页面或失败文件，避免全链路重跑。

### 8.3 提交门禁从“评分”升级为“硬条件”
- `submission_risk_report.json` 关键字段：
  - `hard_gate.passed`
  - `should_block_submission`
  - `blocking_issues`
  - `auto_fix.rounds`
- freeze 阶段与提交队列入口都会强制执行门禁。

### 8.4 新增 API
- `POST /api/projects/{id}/ui-skill/policy-autofix`
  - 按策略建议动作做细粒度修复。
  - 返回 `policy_actions + remaining_blockers`，便于前端展示。

### 8.5 审查材料增强
- 新增/强化：
  - `claim_evidence_matrix.json`
  - `examiner_material_report.json`
  - `examiner_material_sections.md`
- 说明书会自动追加 A03/A04/A05 章节；若模板不兼容，按 `docs/EXAMINER_TEMPLATE_PATCH.md` 手动补丁。
