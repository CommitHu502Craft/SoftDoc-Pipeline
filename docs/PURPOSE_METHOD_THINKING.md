# 软著生成系统设计说明：目的、方法、思考流程

> 适用版本：V2.1+  
> 目标读者：维护者、二次开发者、后续 AI 代理

---

## 1. 这份文档解决什么问题

本项目模块很多、入口很多（CLI/GUI/API），如果只看函数实现，很难快速回答三个核心问题：

1. 系统到底要达成什么业务目标。
2. 为什么当前采用这套实现方法，而不是更简单的模板拼接。
3. 当你要修改某一环节时，应该如何推理影响范围，避免破坏软著交付结果。

本文件给出统一答案，作为“工程决策说明书”。

---

## 2. 项目目的（Purpose）

### 2.1 业务目的

输入项目章程（可由 AI 草拟后人工确认），自动产出软著申请所需的整套材料，并尽量减少人工编辑：

1. 项目规划与页面说明。
2. 可截图的高保真页面。
3. 代码材料与源码 PDF。
4. 操作说明书与辅助填报信息。

### 2.2 质量目的

系统不是“只要能生成就行”，而是追求以下可交付属性：

1. 完整：关键产物缺一不可。
2. 稳定：API 波动、网络异常时尽量不中断全流程。
3. 可控：有明确的回退与兜底，不依赖人工救火。
4. 差异化：降低模板化痕迹，避免批量项目高度同构。

### 2.3 工程目的

1. 支持多入口运行（CLI/GUI/API）。
2. 支持多模型与多网关协议（`chat/completions` 与 `responses`）。
3. 支持可追溯的数据链路（元数据、日志、中间产物）。

## 3. 方法总览（Method）

当前方法不是“单次大模型全包”，而是“分阶段流水线 + 质量闸门 + 降级策略”：

| 阶段 | 输入 | 核心方法 | 输出 | 质量闸门 |
| --- | --- | --- | --- | --- |
| Charter Gate | 项目章程 | 结构化必填校验 + AI 草拟补全 | `project_charter.json` | 业务边界/角色/流程/NFR/验收齐备 |
| Plan | 章程 + 配置 | Master-Detail 规划、行业语义注入 | `project_plan.json` | JSON 可解析、页面结构完整 |
| Spec | Plan + Charter | 规格构建（实体/状态机/权限/API/映射） | `project_executable_spec.json` | 规格结构完整、评审工件生成 |
| HTML | Plan | 母版 + 内容组装、ID 规范化 | `html/*.html` | 页面可渲染、结构不崩 |
| Screenshot | HTML | Playwright 批量渲染截图 | `screenshots/` | 页面截图成功率 |
| Code | Plan + Spec + seeds | 核心文件 AI 闭环改写 + 非核心本地混淆 | `aligned_code/` | 规格已确认、语法/新颖度/禁用片段检测 |
| Verify | 运行产物 | 冒烟 + 关键流程回放 | `runtime_verification_report.json` | 运行通过才允许文档 |
| Document | Plan + 图 + 验证报告 | 模板驱动文档合成 | `*_操作说明书.docx` | 页面和字段可追溯 |
| PDF | 代码目录 | 前后截取策略 + 行数补齐 | `*_源代码.pdf` | 行数规则、分页稳定 |
| Freeze | 全量产物 | 哈希快照 + 复验 + 归档 | `freeze_package/*` | 证据链与复现实验通过 |

## 4. 思考流程（Thinking Process）

本节是维护重点：解释“为什么这样设计”。

### 4.1 为什么是“分阶段”而不是“一次生成全部”

1. 单次大生成失败成本过高，一处失败会拖垮全链路。
2. 分阶段能局部重试，避免重复消耗 token 与时间。
3. 分阶段可插入质量闸门，及时发现问题，不把低质量结果带到后续 PDF/文档。

### 4.2 为什么代码阶段采用“核心 AI + 非核心本地”的混合策略

对应实现：`modules/code_transformer.py`

1. 预算有限时，优先改写对业务指纹影响最大的文件（controller/service/repository）。
2. 通过依赖入度、调用关系、代码规模做优先级画像，减少“改了很多但关键没改”的情况。
3. 非核心文件走本地变换，控制时延与成本，保证批量任务吞吐。

结论：这是“效果-成本-时延”的平衡，而不是追求全文件最强改写。

### 4.3 为什么要做 AI 闭环改写而不是单次改写

对应实现：`modules/code_transformer.py::_rewrite_with_closed_loop`

1. 首轮支持多候选，避免第一版输出直接落盘。
2. 每候选都经过新颖度评分和禁用片段检测。
3. 不达阈值就迭代再改写；长时间无改进则提前停止，防止无效烧 token。
4. 最终仍有语法闸门，语法失败回退，优先保证“可交付”。

### 4.4 为什么源码 PDF 采用“前1500 + 后1500”

对应实现：`modules/code_pdf_generator.py`

1. 该策略贴合软著常见提交习惯，规则稳定、可解释性强。
2. 为提高 AI 改写命中率，代码先排布为“AI 前段 + 其他 + AI 后段”。
3. 当代码不足 3000 行，先尝试纳入 HTML，再不足则循环补齐，优先满足提交页数要求。

### 4.5 为什么 API 客户端要同时支持 transport/protocol 双维切换

对应实现：`core/deepseek_client.py`

1. 不同网关兼容性不同，部分只支持 `chat/completions`，部分支持 `responses`。
2. 网络环境有代理、限流、断流等不确定性，需要请求级重试与退避。
3. 出现 `ProxyError` 可选旁路代理重试，避免全局改环境变量引发副作用。
4. 401/403/额度耗尽属于不可恢复错误，应快速失败，减少无意义重试。

### 4.6 为什么要有结构容错（例如 plan.pages 兼容 dict/list）

对应实现：`modules/spec_builder.py::_iter_pages`

1. 历史数据与不同入口输出结构可能不一致。
2. 不容错会在初始化阶段直接崩溃，导致代码流程完全无法启动。
3. 在边界层统一兼容，可避免在多个模块重复判空和分支。

## 5. 关键参数与建议值

以下参数决定“质量/成本/时长”的平衡，建议在批量任务中保持保守：

1. `novelty_threshold`：过高会导致改写轮次激增，过低会增加同质化风险。
2. `max_rewrite_rounds`：建议小上限（如 2-4），避免时长失控。
3. `rewrite_candidates`：建议首轮 2-3 即可，边际收益递减很快。
4. `http_retries`：建议 2-4；过高会把网络问题放大成超长任务。
5. `retry_max_tokens_cap`：建议开启，重试降载可提高弱网成功率。
6. `max_syntax_fail_files`：建议默认 `0`，语法失败文件不应流入文档/PDF阶段。
7. `min_ai_line_ratio`：建议 `0.15-0.30`，保证 AI 改写代码在总行数里有稳定占比。
8. `enable_embedding_similarity`：默认 `false`；仅在需要更强判重时开启，避免额外时延与模型依赖。
9. `embedding_similarity_weight`：建议不超过 `0.2`，用于补充结构判重而非替代。

## 6. 修改时的影响推理（Change Impact）

如果你修改以下模块，请同步检查关联链路：

1. 改 `project_plan` 字段：同步 `html_generator`、`document_generator`、`auto_submitter`。
2. 改代码文件命名或目录映射：同步 `project_metadata.json` 消费方（尤其源码 PDF 排序）。
3. 改 API 调用协议：同步 GUI/API 配置项、连接测试逻辑、错误提示文案。
4. 改 PDF 行数规则：同步自动提交字段中的源程序量策略与文档描述。
5. 改规格确认策略：同步 Qt/Web/API 三入口交互与后端门禁，避免入口行为漂移。

---

## 7. 运维排障路径（推荐顺序）

1. 先看任务日志首个失败点，不要只看最后报错。
2. 判断是网络/鉴权问题还是内容解析问题。
3. 网络类优先检查：端点、协议、代理、重试参数。
4. 内容类优先检查：JSON 解析失败样本、页面结构、语法闸门回退日志。
5. 代码阶段优先查看 `novelty_quality_report.json` 的 `avg_novelty`、`syntax_fail_count`、`ai_line_ratio`。
6. 仅在必要时提高重试次数；默认优先缩小请求规模而不是盲目重试。
7. 若通过 API 调用流水线，可在请求体 `code_generation_overrides` 中覆盖质量参数，便于批量任务按场景调节阈值。

## 8. 维护原则（给未来 AI/开发者）

1. 先保交付稳定，再追求局部最优。
2. 能在边界层容错，不把复杂判断扩散到全项目。
3. 新增策略必须可解释、可配置、可回退。
4. 修改任何规则时，必须写清楚“目的、方法、影响范围”。
