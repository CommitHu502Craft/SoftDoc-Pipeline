# V3.1 运行时 Skill 与门禁实操指南

本文面向当前代码基线，目标是把“生成可用项目”升级为“生成可审查、可提交、可复核的材料包”。

---

## 1. 目标

V3.1 的核心不是多生成页面，而是把以下三件事打通：

1. 生成决策可追溯：页面怎么规划、为什么这么生成、规则是什么。
2. 审查证据可绑定：每条功能声称都能落到截图/API/代码/回放。
3. 提交门禁可执行：失败不是停在提示，而是自动修复后再判定。

---

## 2. 新增架构层

在原流水线前后新增了三层：

1. Skill Orchestration Layer  
   产出：
   - `ui_skill_profile.json`
   - `ui_blueprint.json`
   - `screenshot_contract.json`
2. Runtime Compliance Layer  
   产出：
   - `runtime_rule_graph.json`
   - `skill_compliance_report.json`
   - `skill_policy_decision_report.json`
3. Hard Gate Layer  
   产出：
   - `submission_risk_report.json`
   - `skill_autorepair_report.json`

## 4. 门禁检查项（硬条件）

当前门禁至少覆盖以下检查：

1. `charter`
2. `naming_consistency`
3. `ui_skill_consistency`
4. `runtime_skill_compliance`
5. `spec_consistency`
6. `claim_evidence`
7. `novelty`
8. `document_screenshot`
9. `timeline_consistency`
10. `evidence_chain`

通过条件：

1. `hard_gate.passed = true`
2. `should_block_submission = false`

## 5. 运行时合规报告说明

文件：`skill_compliance_report.json`

### 5.1 `summary` 关键字段

1. `passed`
2. `rule_pass_ratio`
3. `critical_failed_rules`
4. `failed_rule_ids`
5. `evidence_trace_count`
6. `evidence_preview_count`
7. `evidence_group_counter`

### 5.2 `evidence_trace / evidence_preview`

每条证据会带以下字段：

1. `trace_id`
2. `rule_id`
3. `file_rel`
4. `line`
5. `jump_anchor`（`file:line`）
6. `message`

可直接用于前端定位和修复排查。

---

## 6. 策略裁决报告说明

文件：`skill_policy_decision_report.json`

### 6.1 `summary.auto_fix_actions`

可自动执行的动作列表（按优先级排序）。

### 6.2 `summary.action_resolution`

每个动作包含：

1. `action`
2. `priority_index`
3. `dimensions`
4. `critical_rule_count`
5. `rule_count`
6. `rule_ids`

用于解释“为什么先修这个”。

---

## 7. 自动修复策略

### 7.1 总体自动修复

入口：`submission-risk/check`（`enable_auto_fix=true`）

行为：门禁失败时执行 `run_submission_auto_fix()`，按轮次自动修复并复检。

### 7.2 策略动作自动修复（更细粒度）

入口：`POST /api/projects/{id}/ui-skill/policy-autofix`

行为：

1. 先读取策略建议动作
2. 调用 `run_skill_autorepair(policy_actions=...)`
3. 仅执行命中动作对应的修复范围（copy/html/action）
4. 返回 `policy_actions + remaining_blockers`

## 8. 审查材料 A03/A04/A05

### 8.1 生成路径

1. `document_generator` 在说明书阶段编译声称-证据矩阵
2. 生成审查报告与章节：
   - `examiner_material_report.json`
   - `examiner_material_sections.md`
3. 自动将 A03/A04/A05 追加到说明书正文

### 8.2 你仍需关注的手工项

1. 若模板结构与占位符不一致，按 `docs/EXAMINER_TEMPLATE_PATCH.md` 修补模板。
2. 若模板解析出错，优先使用纯文本循环占位而不是复杂表格标签。

## 9. 一次通过操作剧本（推荐）

### 9.1 准备

1. 补全章程（全称/简称/术语字典必填）。
2. 使用 Skill Studio 指定领域、页面数、功能偏好与 token 策略。

### 9.2 生成

1. 跑主流水线。
2. 查看策略裁决卡片：
   - `critical_failed_rules`
   - `policy_auto_fix_actions`

### 9.3 修复

1. 点击“执行策略修复”（或调接口 `ui-skill/policy-autofix`）。
2. 再执行 `submission-risk/check`。

### 9.4 提交前判定

确认：

1. `submission_risk_report.json` 中 `hard_gate.passed=true`
2. `should_block_submission=false`
3. `examiner_material_report.json` 中 `passed=true`

---

## 10. 常见失败与定位

### 10.1 `claim_evidence_matrix passed=false`

优先看：

1. `binding_ratio`
2. `unbound_claims`
3. `hard_blocking_issues`

处理顺序：

1. 先补 HTML 锚点与截图契约覆盖
2. 再重跑 `verify -> document`

### 10.2 说明书阶段失败并提示 auto-fix 未收敛

查看：

1. `submission_risk_report.json`
2. `skill_autorepair_report.json`
3. `examiner_material_report.json`

常见根因：

1. 文档声称过度，超出代码/API/回放能力
2. 页面截图覆盖不足
3. 规则关键项（如按钮动作、showModal 序列）未达标

## 13. 注意事项

1. 运行时技能（`runtime_skills/`）是给“生成系统”用，不是给 Codex 本身用。
2. `runtime_skill_override.json` 现在是严格 schema 校验，非法键会阻断覆盖应用。
3. 提交门禁不会因为评分高而放行，硬条件失败即阻断。
